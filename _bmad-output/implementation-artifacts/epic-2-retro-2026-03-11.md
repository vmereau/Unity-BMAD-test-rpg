## Successes & Strengths

### 1. Complete Combat Loop Delivered
All planned Epic 2 mechanics are in place and functional: stamina resource management, 3-hit timed combo, manual block, perfect block timing window, dodge roll with i-frames, enemy patrol/engage/attack AI, and a full health-and-death loop. The combat prototype is playable end-to-end.

### 2. Edit Mode Test Foundation Is Solid
Tests grew from 4 to 57 across the epic. Every story added tests covering its core logic formula. The pattern is consistent: pure-math tests, no MonoBehaviour lifecycle dependencies. This gives strong regression safety as `PlayerCombat` grew from a stub to a complex state machine.

### 3. CLAUDE.md as Living Documentation
The team consistently read and updated CLAUDE.md throughout the epic. Critical discoveries (InputSystem_Actions dual-edit, AnimatorController conditionMode bug, OnDisable null guard) were documented immediately and prevented repeat mistakes in subsequent stories. By Story 2.7 the `InputSystem_Actions.cs` dual-edit pattern was second nature.

### 4. Code Review Catching Real Issues
Every story had substantive code review fixes applied — not formatting, but actual correctness issues (NullReferenceException risk in `PlayerStateManager` setters, `IsAttacking` stuck-true on Consume fail-path, block-cancel guard missing from `OnBlockCanceled`). The adversarial code review workflow is paying off.

### 5. Player State Manager Architectural Refactor (Story 2.6)
Story 2.6 correctly identified that `PlayerCombat` was accumulating too many responsibilities and introduced `PlayerStateManager` as a clean separation of concerns. The refactor delivered atomicity (trigger + state in a single `SetAttacking()` call) and a unified source of truth for gating logic. The `CanAttack()`, `CanBlock()`, `CanDodge()` helpers in `PlayerStateManager` are a significant quality improvement.

## Challenges & Growth Areas

### 1. Design Pivot: Directional Attacks → Timed Combo (Story 2.3)
**Pattern:** Mid-epic scope change required a full story replacement.
The directional attack system (Story 2.3-directional) was fully specified and implemented, then superseded by a timed combo design that better matched the Gothic-style combat feel. Two stories were written for Story 2.3 — one superseded.
**Impact:** Added complexity to sprint-status tracking. The `superseded` status is non-standard.
**Lesson:** Design decisions on core mechanic feel (directional vs. timed) should be validated with a quick paper prototype or by examining the reference game more closely before committing to a full story. A spike/prototype story is better than a superseded story.

### 2. InputSystem_Actions.cs Dual-Edit Pattern
**Pattern:** Every story that added a new input action (2.4 Block, 2.7 Dodge) had the same issue: Unity auto-regenerated `InputSystem_Actions.cs` at `Assets/` root after editing `.inputactions`, creating a ghost file. Caught and deleted in stories 2.4 and 2.7.
**Impact:** Minor — no production bugs, but adds a cleanup step to every input-change story.
**Lesson:** Now documented in CLAUDE.md as a mandatory dual-edit pattern. Future epics should treat this as a known workflow, not a surprise.

### 3. Play Mode Validation Consistently Left Unchecked
**Pattern:** Stories 2.3, 2.5, 2.7, 2.8, and 2.9 all had their Play Mode validation task unchecked/pending manual verification.
**Impact:** The full combat loop has not been manually verified by a human running the game. All logic is unit-tested but the feel, timing, and visual correctness of animations, hit detection, and the enemy AI state machine are untested in a live play session.
**Lesson:** Play Mode validation cannot be skipped indefinitely. Before Epic 3 begins, a play session should be run to verify the full combat loop end-to-end.

### 4. AnimatorController Complexity Grew Without Visual Validation
**Pattern:** Animator states and transitions were added via YAML edits and MCP tool calls, with a known `conditionMode` bug on bool transitions.
**Impact:** Transitions may not behave correctly in the Unity Animator window. `WriteDefaultValues: 0` was fixed for some states but there were multiple passes (Story 2.9 fixed 4 states that were still at `m_WriteDefaultValues: 1`).
**Lesson:** Animator changes need a human to open the Animator window and verify the state machine visually. This should be a checklist item for any story touching `.controller` files.

### 5. Story 2.9 Dev Notes Are Thin
Story 2.9's `Dev Agent Record` has empty `Completion Notes` and `Debug Log References`, compared to the thorough notes in earlier stories. This makes it harder to understand what trade-offs were made and why.
**Lesson:** Completion notes are valuable retrospective material. The team should maintain thoroughness even in later stories when momentum is high.

## Key Insights & Learnings

1. **Combat state machines benefit from an aggregator component** — `PlayerStateManager` (Story 2.6) was not in the original epic plan but emerged as a natural need when `PlayerCombat` reached ~300 lines. Anticipate this earlier in similar epics.

2. **"The GDD vs. the implementation" gap** — Dodge was specified as "Space + WASD direction" in the GDD but Space was already Jump. Left Ctrl was used as a pragmatic fix. The GDD should be reviewed for input conflicts before an epic starts, not discovered mid-implementation.

3. **The combo window delay (Phase 1) emerged from implementation** — The two-phase window (delay then open) was not in the original Story 2.3 spec, but the dev agent added it for better game feel. This divergence from spec produced a better mechanic. A retrospective note that the spec underspecified timing feel is valuable for future story authors.

4. **NavMesh + AI setup has external requirements** — Story 2.8 required NavMesh baking, a `Player` tag, physics layers, and an Editor-only `NavMeshBaker.cs`. These prerequisites should be listed explicitly in future AI-related stories.

5. **BlockGateTests tests local helpers, not the real API** — Flagged in Story 2.4 code review (action item: "flag for Epic 2 retrospective"). Tests that exercise local formula helpers rather than the actual `StaminaSystem.HasEnough()` will miss real regressions. This debt should be addressed.
